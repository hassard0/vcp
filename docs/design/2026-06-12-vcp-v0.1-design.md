# VCP v0.1 — Design Rationale

**Date:** 2026-06-12
**Status:** Design accepted, in implementation
**Author:** hassard0

This document records *why* VCP is shaped the way it is. The normative artifact is
[`SPECIFICATION.md`](../../SPECIFICATION.md); this is the design narrative behind it.

## Problem

MCP's breakthrough is ecosystem simplicity — host/client/server, JSON-RPC, and a
small set of primitives (tools, resources, prompts, sampling, roots, elicitation).
That same easy composability is its security weakness: the MCP spec itself states
that MCP enables arbitrary data-access and code-execution paths and *cannot enforce
many security principles at the protocol level*. Authorization is **optional**.
Tool descriptions are trusted by the model. Local servers run with the host's full
ambient authority.

The documented failure modes are concrete and exploited in the wild:

- **Tool poisoning / rug pulls** — hidden instructions in tool descriptions; tool
  definitions mutate *after* approval.
- **Over-trusted local servers** — STDIO servers run with the client's privileges.
- **Token passthrough / confused deputy / SSRF / session hijacking.**
- **Stateful-session ambiguity** — sticky sessions, load-balancing friction.

## Thesis

**VCP is MCP plus zero-trust capability execution.** Security, provenance, policy,
and determinism are *protocol requirements*, not implementation advice. The model is
never the authority — it is only a planner. A signed **Gateway** is the
authority-enforcing execution layer.

No integration standard can have *no* weaknesses. The goal is narrower and
achievable: make each known MCP failure mode **impossible, non-compliant,
auditable, or safely contained.**

## Decisions (this design)

| Decision | Choice | Rationale |
|---|---|---|
| Spec repo | `hassard0/vcp` | Owns the bare protocol name; spec is source of truth. |
| Impl repo | `hassard0/vcp-servers` | Reference SDKs + gateways pin spec by hash. |
| Languages | TS, Python, Go, Rust | Where the agent ecosystem actually builds. |
| Variants | light = client/SDK + MCP bridge; heavy = full gateway | Adoption path + reference enforcement. |
| Spec scope | tight normative core + Open-RFC backlog | Ship the enforceable core; don't over-commit speculative parts. |
| Conformance | shared language-agnostic JSON test vectors | "VCP-compliant" must be mechanically checkable across all four langs. |

## Improvements over the source draft (learned from MCP)

1. **Canonical JSON pinned to RFC 8785 (JCS) + SHA-256.** The draft said
   `canonical_json` but never defined it. Ambiguous canonicalization is itself an
   attack surface (hash-confusion). Made normative.
2. **Grants are concrete proof-of-possession tokens**, not loose JSON — short-TTL,
   key-bound (DPoP-style), audience = capability hash, bound to `plan_hash` +
   `argument_hash`, and *attenuable-only* (a gateway can narrow, never widen).
3. **Structured, remediable denials** — borrowed from `mcp-ledger`'s
   `LEDGER_LIMIT_EXCEEDED`: every deny is machine-actionable so a planner can
   self-correct instead of looping.
4. **Bounded results + pagination via state handles** — unbounded tool output
   flooding the model context is a real, underspecified MCP pain.
5. **Elicitation & sampling routed through the gateway** as policy-gated, redacted,
   budgeted *delegated-inference* capabilities — closing MCP's sampling injection
   surface.
6. **Cancellation revokes the grant** — not a best-effort notification.
7. **Ecosystem cross-references**: `cani` as a conformant Policy Authority / PDP;
   `mcp-ledger` as a conformant audit + budget substrate (grants carry a budget,
   the ledger enforces it); list VCP in `prosecco-ai-standards`.

## Explicitly out of scope for v0.1 (→ RFCs)

A hosted transparency registry, a shipped sandbox runtime (gVisor/Firecracker),
reproducible-build provenance, formal policy verification, and DLP/data-flow
proofs. We *specify the profiles and enforce allowlists*; the heavy runtime work is
deferred to discussion.

## Build order

1. `vcp` spec repo (SPECIFICATION + schemas + governance + RFC backlog + Discussions).
2. `vcp-servers`: conformance vectors → TS (canonical) → Python → Go → Rust, light then heavy, the four ports built in parallel.
